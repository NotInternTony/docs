## How Simulation Works

In sandbox, use special amounts or metadata flags to force specific payment outcomes. This lets you test every branch of your payment handling logic without waiting for real blockchain confirmations.

## Test Scenarios

Use these magic amounts when creating a payment session in sandbox to trigger specific outcomes:

| Amount | Outcome | Status | Description |
| --- | --- | --- | --- |
| `1.00` | Success | `completed` | Payment confirms immediately |
| `2.00` | Delayed success | `completed` | Payment confirms after ~30 seconds |
| `3.00` | Pending | `pending` | Payment stays pending indefinitely |
| `4.00` | Failed | `failed` | Transaction fails on-chain |
| `5.00` | Insufficient funds | `failed` | Simulates wallet with insufficient balance |
| `6.00` | Expired | `expired` | Session expires before payment |
| `7.00` | Network timeout | `failed` | Simulates a network timeout error |
| `0.01` | Duplicate | `failed` | Simulates a duplicate payment attempt |

## Using Magic Amounts

```typescript
// Server-side: create a session that will simulate a failure
const session = await grain.paymentSessions.create({
  amount: "4.00",  // Forces a failed transaction
  currency: "USDC",
  chain: "ethereum",
  metadata: { orderId: "test_fail_001" },
});
```

## Using Metadata Flags

For more granular control, pass a `simulateOutcome` field in your session metadata.

```typescript
const session = await grain.paymentSessions.create({
  amount: "25.00",
  currency: "USDC",
  chain: "ethereum",
  metadata: {
    orderId: "test_order_002",
    simulateOutcome: "delayed_success",  // Override the magic amount behavior
  },
});
```

### Available `simulateOutcome` Values

| Value | Behavior |
| --- | --- |
| `success` | Immediate confirmation |
| `delayed_success` | Confirms after ~30 seconds |
| `pending` | Stays in pending state |
| `failed` | On-chain failure |
| `insufficient_funds` | Wallet balance error |
| `expired` | Session expires |
| `network_timeout` | Network timeout error |
| `rejected` | User rejects in wallet |

<Note>
  When `simulateOutcome` is set in metadata, it takes priority over magic amounts.
</Note>

## Verify Results

After running a simulation, check the outcome in three places:

1. **Client callbacks** — Confirm your `onSuccess` / `onError` / `onExpired` handlers fire correctly.
2. **Webhooks** — Verify the corresponding webhook event is delivered to your endpoint.
3. **Dashboard** — Review the payment in **Payments** on the sandbox dashboard.

<Tip>
  Combine simulated outcomes with your automated test suite to cover all payment paths in CI.
</Tip>
