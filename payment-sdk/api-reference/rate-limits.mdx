## Rate Limit Tiers

Grain enforces rate limits per API key to ensure platform stability. Limits vary by plan.

| Plan | Requests per Minute | Requests per Day | Burst (per second) |
| --- | --- | --- | --- |
| Free | 60 | 10,000 | 5 |
| Growth | 300 | 100,000 | 20 |
| Enterprise | 1,000 | Unlimited | 100 |

<Note>
  Rate limits apply per API key, not per endpoint. Both publishable and secret keys have independent limits.
</Note>

## Rate Limit Headers

Every API response includes headers indicating your current rate limit status.

| Header | Description |
| --- | --- |
| `X-RateLimit-Limit` | Maximum requests allowed in the current window |
| `X-RateLimit-Remaining` | Requests remaining in the current window |
| `X-RateLimit-Reset` | Unix timestamp (seconds) when the window resets |
| `Retry-After` | Seconds to wait before retrying (only on `429` responses) |

```
HTTP/1.1 200 OK
X-RateLimit-Limit: 300
X-RateLimit-Remaining: 247
X-RateLimit-Reset: 1742486460
```

## Handling 429 Responses

When you exceed the rate limit, the API returns a `429 Too Many Requests` response.

```json
{
  "error": {
    "code": "rate_limited",
    "status": 429,
    "message": "Rate limit exceeded. Retry after 12 seconds.",
    "details": {
      "retryAfterMs": 12000
    }
  }
}
```

## Implement Exponential Backoff

<Tip>
  Always implement exponential backoff with jitter for retry logic. This prevents thundering herd problems when multiple clients retry simultaneously.
</Tip>

```typescript
async function requestWithRetry(
  fn: () => Promise<Response>,
  maxRetries = 5
): Promise<Response> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const response = await fn();

    if (response.status !== 429) {
      return response;
    }

    const retryAfter = response.headers.get("Retry-After");
    const baseDelay = retryAfter
      ? parseInt(retryAfter, 10) * 1000
      : Math.pow(2, attempt) * 1000;

    // Add jitter (0-50% of base delay)
    const jitter = Math.random() * baseDelay * 0.5;
    const delay = baseDelay + jitter;

    console.warn(`Rate limited. Retrying in ${Math.round(delay)}ms (attempt ${attempt + 1}/${maxRetries})`);
    await new Promise((resolve) => setTimeout(resolve, delay));
  }

  throw new Error("Max retries exceeded");
}
```

## Monitor Your Usage

Track your rate limit consumption using the response headers.

```typescript
async function monitoredRequest(url: string, options: RequestInit) {
  const response = await fetch(url, options);

  const remaining = parseInt(response.headers.get("X-RateLimit-Remaining") || "0", 10);
  const limit = parseInt(response.headers.get("X-RateLimit-Limit") || "0", 10);

  if (remaining < limit * 0.1) {
    console.warn(`Rate limit warning: ${remaining}/${limit} requests remaining`);
  }

  return response;
}
```

## Best Practices

- **Cache responses** when possible to reduce unnecessary API calls.
- **Batch operations** instead of making individual requests in tight loops.
- **Use webhooks** for payment status updates instead of polling the Get Payment endpoint.
- **Queue requests** server-side to smooth out traffic spikes.
- **Monitor the `X-RateLimit-Remaining` header** and throttle proactively before hitting the limit.

<Warning>
  Repeated rate limit violations may result in temporary key suspension. If you consistently hit limits, consider upgrading your plan or contacting support.
</Warning>
